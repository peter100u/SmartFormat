# Mova 产品架构文档

## 产品定位

Mova 是一款面向普通用户的现代、干净、低价、移动端优先的格式工厂。它应该比传统格式转换工具更精致，比专业转码软件更简单，也比在线转换网站更安全，因为 MVP 阶段的媒体处理都在用户设备本地完成。

产品目标不是打败所有专业媒体工具，而是覆盖足够常见的格式转换、压缩、多文件输入和任务记录场景，成为一款可以支撑一人公司持续经营的日常工具。

## 核心优势

- 使用 Flutter 构建，优先覆盖 iOS 和 Android，后续可扩展到桌面端。
- 基于 `ffmpeg-kit-extended` 提供媒体处理能力。
- 一人公司运营成本低，可以用低价策略切入市场。
- 界面干净，不做捆绑软件，不做激进广告。
- 相比传统付费转换套件，价格更低、体验更轻。

## 产品原则

- 先搭建完整工具货架，再逐步补齐每个工具项。
- 先聚焦媒体格式，不急着扩展到文档、OCR 或云端工作流。
- 基础单文件功能永久免费。
- 后续商业化能力围绕效率、多文件处理、专业效果和高级质量，不从用户手里收回基础能力。
- 功能名称使用用户听得懂的任务语言，避免一上来暴露复杂编码参数。
- 每个任务都必须清楚展示输出位置、处理进度、结果状态和失败原因。

## 当前评审结论

这份文档的主方向是合理的，尤其是以下几点判断是对的：

- 用移动端优先切入，而不是一开始同时做桌面和移动。
- 用“首页 + 任务 + 设置”的底部导航承载产品骨架。
- 用“输入文件集合 + 工具参数 + 输出目标 + 任务记录”作为统一模型。
- 不单独设计“批量任务”，而是让多文件输入自然生成多条任务记录。
- 先做本地处理，不急着做账号、云盘、登录和复杂商业化。

从产品文档角度看，原稿里有 4 个常见风险，需要明确修正：

- 一些“后续能力”混进了 MVP 表达，例如首页最近任务入口、设置里的联系支持和评价 App。
- 一些动作文案和当前 scope 不一致，例如“重新保存”“保存位置”，会让人误以为首版已经支持保存到相册或系统文件。
- 一些技术文档和产品文档还没有完全对齐，例如路由和服务层里仍有后续功能残留。
- 某些描述对非产品背景的人不够友好，容易把“能力列表”和“真正要先做什么”混在一起。

因此这次文档调整的目标不是增加更多内容，而是把 MVP 写得更窄、更一致、更容易继续维护。

## 视频处理

视频是最重要的分类。它最容易让用户感知产品价值，也覆盖了大量 FFmpeg 的常见能力。

### 首批工具项

- 格式转换
- 视频压缩
- 视频转音频

### 后续工具项

- 视频转 GIF
- 视频转 Live Photo
- 视频裁剪
- 视频合并
- 去除音频
- 替换音频
- 提取封面
- 字幕烧录
- 字幕转视频字幕
- 不重新编码修改封装格式
- 平台预设：B 站、抖音、小红书、微信视频号、YouTube、Instagram

### 备注

视频转 GIF 和视频转 Live Photo 都属于高感知价值功能，但实现和调试成本高，不进入 MVP。后续如果产品验证成立，再作为高价值工具补充。

## 音频处理

### 首批工具项

- 格式转换
- 音频压缩
- 视频转音频

### 后续工具项

- 音量标准化

### 后续工具项

- 调整图片尺寸
- WebP 转换

## MVP 范围

第一版可用产品应该有完整产品壳，并且每个主要分类都有少量真正可用的工具。

### MVP 完整闭环

MVP 只有在用户能完整跑通一次真实处理任务时，才算成立：

用户可感知的最短路径应该尽量短：

```text
打开 App
-> 选择工具
-> 选择文件
-> 确认参数
-> 开始任务
-> 查看结果
```

系统内部仍然要完成完整闭环：

```text
选择工具
-> 选择输入文件集合
-> 自动预检文件
-> 带出默认参数和默认输出目标
-> 用户只在需要时调整参数
-> 创建任务记录
-> 执行任务并展示进度
-> 进入结果页
-> 支持打开、分享、再次处理、重试或反馈问题
-> 后续在任务记录中找回
```

这个闭环比工具数量更重要。少量可靠工具，比一堆无法清楚产出、保存和找回结果的入口更有价值。

减少用户操作路径的原则：

- 不把“文件预检”设计成单独确认步骤，除非文件不兼容、权限不足或空间不足。
- 不让用户每次都选择输出目标，MVP 默认输出到 App 内结果，任务完成后再让用户分享。
- 常见参数必须有默认值，例如格式转换默认推荐格式、压缩默认推荐质量。
- 高级参数默认折叠，用户不设置也能完成任务。
- 结果页只突出最重要动作：打开、分享、再处理一个。任务记录入口保持可见但不抢主操作。

Mova 的核心处理模型应该始终保持一致：

```text
输入文件集合 + 工具参数 + 输出目标 -> 处理任务 -> 结果页和任务记录
```

选择 1 个输入文件时生成 1 条任务记录；选择多个输入文件时，每个文件生成 1 条任务记录。多文件不是单独的产品入口，也不需要拆成独立的“批量任务”概念。

### MVP 功能清单

- 首页包含视频、音频、图片分类 Tab。
- 底部导航包含首页、任务、设置。
- 视频格式转换。
- 视频压缩。
- 视频转音频。
- 音频格式转换。
- 音频压缩。
- 图片格式转换。
- 图片压缩。
- HEIC 转 JPG/PNG。
- 任务记录，包含处理中、成功、失败、取消状态。
- 任务记录，包含输入文件、输出文件、工具类型、时间、状态和快捷操作。
- 输入源选择。
- 默认输出目标。
- 文件预览和预检。
- 权限请求和权限拒绝后的恢复引导。
- 任务完成后的结果页。
- 用户能看懂的错误提示。
- 临时文件和缓存清理。
- 简短的首次启动说明。
- 设置页反馈入口和失败任务反馈入口。
- 隐私政策、用户协议和设置页「许可证」入口。
- 「许可证」入口链接到 FFmpegKit、FFmpeg、第三方组件、构建信息和源码获取说明。

### MVP 输入源

输入源表示文件从哪里来。它应该独立于工具和输出目标，并且默认支持单文件或多文件选择。处理任务数量由输入文件数量决定，不需要设计独立的“批量任务”入口。

首批输入源：

- 相册。
- 系统文件选择器。

输入数量规则：

- 选择 1 个文件时，生成 1 条任务记录。
- 选择多个文件时，每个输入文件生成 1 条任务记录。
- 多文件输入使用同一套工具参数和输出目标。
- 某个文件失败时，不影响其他文件继续处理。

暂缓输入源：

- 从其他 App 分享进入。
- 最近文件或最近任务记录。
- 网盘导入。
- 网络 URL 导入。
- 相机拍摄。
- 屏幕录制。

其他 App 分享进入和最近文件/任务记录都很有价值，但它们会增加平台适配、路径恢复和状态管理复杂度。MVP 先只实现相册和系统文件选择器。网盘应该暂缓，因为它会带来账号登录、授权、传输进度、失败重试、隐私和客服支持复杂度。

### MVP 输出目标

输出目标表示处理后的文件去哪里。它应该独立于工具和输入源。

首批输出目标：

- App 内结果存储。
- 分享到其他 App。

暂缓输出目标：

- 保存到相册。
- 保存到系统文件。
- 网盘导出。
- 自动导出规则。
- 监听文件夹输出。

MVP 先保证所有成功任务都有 App 内结果，并能通过系统分享面板交给微信、邮件、网盘或其他 App。App 内结果存储是默认输出目标，不要求用户在任务开始前手动选择。保存到相册和保存到系统文件都很重要，但平台权限、路径选择、覆盖策略和审核细节更复杂，先放到后续小样验证后再进入正式范围。

每个已完成任务都必须告诉用户输出文件存在哪里。如果分享失败，只要可能，用户仍然应该能找到 App 内结果，并重新分享或重新处理。

### 文件预检

处理开始前，App 应该检查所选文件。预检默认在后台完成，不应该变成用户每次都必须确认的一页。

正常情况只展示摘要：

- 文件名。
- 文件大小。
- 当前格式。
- 预计输出格式。
- 主要参数摘要。

展开详情时可展示：

- 文件名。
- 文件类型。
- 文件大小。
- 视频/音频时长。
- 视频/图片分辨率。
- 当前格式或封装。
- 预计输出格式。
- 基础兼容性状态。

以下情况应阻止处理或给出警告：

- 文件类型不支持。
- 文件无法读取。
- 所选输出目标不可用。
- 可用存储空间可能不足。
- 缺少必要权限。

### 权限流程

权限是产品流程的一部分，不是纯技术细节。

MVP 权限场景：

- 读取相册。
- 从系统存储选择文件。
- 长任务通知权限，可选。

如果用户拒绝权限，App 应该解释为什么需要该权限，并提供跳转系统设置的路径。用户不能被卡在一个死页面上。

### 结果页

任务完成后，用户应该看到清晰的结果页或结果弹层。

结果页应展示：

- 输出文件名。
- 输出大小。
- App 内结果位置。
- 使用的工具。
- 处理状态。
- 打开。
- 分享。
- 再次处理。
- 再处理一个文件。
- 查看任务记录。

结果页负责闭环。它告诉用户发生了什么，结果在哪里，接下来能做什么。

### 错误处理

FFmpeg 错误需要翻译成用户能理解的提示。

首批错误分类：

- 格式不支持。
- 文件无法读取。
- 文件可能已损坏。
- 权限不足。
- 存储空间不足。
- 输出目标失败。
- 编码器或解码器不可用。
- 任务已取消。
- 未知处理失败。

每个失败任务都应该展示简短解释，并至少提供一个下一步操作：

- 重试。
- 选择其他输出格式。
- 重新分享或更换输出目标。
- 反馈此问题。
- 删除任务记录。

### 存储和缓存

媒体处理很容易占用大量空间，因此 MVP 必须把存储行为讲清楚。

App 应该区分：

- 临时处理文件。
- App 结果文件。
- 任务记录元数据。

设置页应包含清理缓存。清理缓存后任务记录应继续存在，但如果输出文件已经不存在，任务记录要标记文件缺失。

### 首次启动

首次启动说明要短，并建立信任感。

需要说明：

- MVP 阶段媒体处理发生在设备本地。
- 基础功能免费。
- 结果可以后续在任务记录中找回。

不要做很长的新手引导。用户应该能快速开始处理文件。

### 反馈

反馈入口应同时出现在设置页和失败任务记录中。

从失败任务反馈时，App 可以附带：

- App 版本。
- 设备型号。
- 系统版本。
- 工具类型。
- 错误分类。
- FFmpeg 错误摘要。

除非用户明确选择附加文件，否则 App 不应上传用户的输入文件或输出文件。

### 法务和信任

公开发布前，MVP 应包含清晰可见的法务与信任入口：

- 隐私政策。
- 用户协议。
- 许可证。
- FFmpegKit、FFmpeg 和相关编解码器许可说明。
- 当前构建信息和源码获取方式。
- 本地处理说明。

这点很重要，因为 App 会处理用户的私密媒体文件。

发布前门槛：

- 未确认 FFmpegKit 实际构建类型、GPL/nonfree 状态、启用组件和第三方许可证前，不应对外 GA。
- 未准备许可证文本、构建信息、源码获取方式和隐私政策前，不应提交应用商店审核。
- 未确认本地处理说明和反馈数据范围前，不应接入会上传用户文件的功能。

许可证页面的具体信息架构、页面文案和发布前检查清单见 [Mova 许可证披露方案](./license-disclosure.md)。

### 暂缓进入 MVP 的能力

- PDF 和 Office 转换。
- OCR。
- 云端上传或云端转码。
- 完整视频剪辑时间线。
- 屏幕录制。
- AI 超分辨率。
- 团队账号。

## 免费和后续商业化策略

Mova 早期应该足够慷慨，但商业边界要从第一天就清楚。

MVP 不需要登录注册，不需要账号系统，也不需要付费页、恢复购买或权益管理入口。第一版先把产品闭环和口碑做出来，商业化能力只保留在路线图中，不进入 MVP 设置页。

### 永久免费

- 单文件视频转换。
- 单文件音频转换。
- 单文件图片转换。
- 基础视频压缩。
- 基础图片压缩。
- 视频转音频。
- 常用输出预设。

### 后续可考虑收费的高级能力

- 多文件并发处理。
- 多文件输出命名模板。
- 视频转 Live Photo。
- 音频降噪。
- 字幕烧录。
- 硬件加速控制。
- 自定义预设。
- 文件夹自动监听。
- 高级输出命名。

### 对用户的表达

Mova MVP 可以先开放全部已完成能力免费使用。后续如果产品验证成立，再考虑把多文件效率、高级预设和高级质量能力做成低价高级版。不要把已经承诺永久免费且用户高频依赖的基础功能改成收费。

## 移动端 UI 结构

### 首页

首页应该展示完整工具套件，同时适合移动端单手操作：

- 顶部区域：App 名称、任务记录入口和设置入口。
- 主区域：视频、音频、图片分类 Tab。
- 工具列表：当前分类 Tab 下的可滚动卡片列表。
- 快捷入口：可选，固定展示最常用工具，例如视频转换、视频压缩、视频转音频、图片转换。
- 任务记录入口：有任务运行时可展示紧凑进度提示。

首页路径优化：

- 高频工具可以放在分类 Tab 上方或当前分类的前几位，减少用户滑动。
- 工具卡片点击后直接进入文件选择，不先进入说明页。
- 工具说明放在卡片副标题和参数页里，不单独做教程页。
- 有任务处理中时，首页只展示轻量进度条或状态入口，不强行跳转任务页。

### 底部导航

移动端底部导航只放全局页面：

- 首页
- 任务
- 设置

视频、音频、图片不放在底部导航里。它们属于首页内部的工具分类，而不是 App 级页面。

### 任务页

任务页只展示任务记录。每条记录通过状态字段表达当前进度，不再拆分额外页面或 Tab。

任务记录状态：

- 处理中。
- 成功。
- 失败。
- 已取消。

任务记录展示：

- 原始输入文件名和来源。
- 输出文件名和 App 内结果位置。
- 使用的工具，例如视频压缩或图片转换。
- 创建时间。
- 处理状态。
- 处理中任务的进度。
- 失败任务的错误原因和重试入口。
- 基础参数摘要，例如输出格式或压缩等级。
- 快捷操作：打开、分享、重新分享、重试、删除记录。

任务记录不能被当成永久文件备份。如果用户在 App 外删除了输出文件，App 应显示文件已不存在，并提供删除记录或在原输入仍可用时重新处理。

### 设置页

设置页是必需的一级页面，因为它承载信任、支持、多语言、法务和基础偏好设置。

首批设置项：

- 多语言。
- 意见反馈。
- 用户协议。
- 隐私政策。
- 许可证。
- 关于 Mova。
- App 版本。
- 清理缓存。

后续设置项：

- 主题模式。
- 通知偏好。
- 默认处理质量。
- 默认输出位置。
- 联系支持。
- 评价 App。

MVP 设置页应保持简单，并按分组展示：

```text

支持
- 意见反馈

隐私与法律
- 隐私政策
- 用户协议

- 清理缓存

- 关于 Mova
```

### 首页分类 Tab

首页使用分类 Tab：

```text

```

当前选中的 Tab 下方展示垂直 `ListView` 工具卡片。每张卡片包含：

### 工具详情页

每个工具遵循同一套流程：

```text
选择文件
-> 确认推荐参数
-> 开始任务
-> 查看进度和结果
```

默认参数应该能直接开始任务。高级设置默认折叠，输出目标默认使用 App 内结果，任务完成后再提供分享。

工具详情页路径优化：

- 用户从工具卡片进入后，应优先弹出文件选择器或展示醒目的“选择文件”主按钮。
- 文件选择完成后，在同一页展示文件摘要、推荐参数和开始按钮，不拆成多页。
- 参数选择使用预设优先，例如“推荐、较小体积、更高清晰度”，高级编码参数放到折叠区。
- 只有文件不支持、权限不足、空间不足或参数冲突时，才打断用户流程。
- 多文件输入时只设置一次参数，不让用户逐个文件重复确认。

### 任务记录模型

任务记录应该是产品的一等能力。正在处理、处理成功、处理失败都进入同一个记录列表。

必需状态：

- 处理中
- 成功
- 失败
- 已取消

每条任务记录都应该展示输入文件、输出格式、处理进度、App 内结果位置，以及失败时清晰的错误说明。

## 技术产品说明

- 所有工具共用一套任务模型。
- 每个工具先生成结构化处理请求，再转换成 FFmpeg 命令。
- 当前开发阶段的媒体处理能力规划基于 `ffmpeg_kit_extended_flutter` 的 `base` 小包构建，当前目标配置为 `gpl: false`；发布包必须记录实际构建类型、版本、GPL 状态和第三方组件清单。
- 避免把原始 FFmpeg 命令字符串散落在 UI 文件里。
- 命令生成、任务执行、进度解析、UI 状态应分离。
- 设计工具注册表，让新增工具项主要是配置和命令生成，而不是重写导航。
- 输入源、处理工具、输出目标应相互独立，通过任务系统连接。
- 任务记录保存元数据，不等同于永久文件备份。
- App 内「许可证」页面必须展示 FFmpegKit、FFmpeg、第三方组件、许可证文本、构建信息和源码获取方式，详见 [许可证披露方案](./license-disclosure.md)。

## 成功指标

- 新用户在 10 秒内能理解 Mova 是本地媒体转换工具。
- 首次任务完成率：新用户第一次选择工具后，能成功完成至少 1 个处理任务。
- 处理成功率：已开始的任务大多数能进入成功状态，失败任务必须有明确错误分类。
- 结果找回率：用户能从结果页或任务记录找到输出文件和参数摘要。
- 结果分享率：成功任务中有一部分用户会点击打开或分享，说明输出闭环有效。
- 7 日内二次使用率：用户在一周内再次打开 App 并完成任务。
- 失败任务反馈率：失败任务能引导用户反馈问题，而不是直接流失。
- 新增工具项主要依赖配置和命令生成，不需要改动整体 App 架构。
